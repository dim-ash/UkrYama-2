1 <h3>Is the accessRules structure going to change?</h3>
2 <p>In you old controllers you had an accessRules method to provide access restrictions to your actions.</p>
3 
4 <div class="code"><code>
5  public function accessRules()
6  {
7  return array(
8  array('allow', // allow all users to perform 'index' and 'view' actions
9  'actions'=&gt;array('index','view'),
10  'users'=&gt;array('*'),
11  ),
12  array('allow', // allow authenticated user to perform
13  // 'create' and 'update' actions
14  'actions'=&gt;array('create','update'),
15  'users'=&gt;array('@'),
16  ),
17  array('allow', // allow admin user to perform 'admin'
18  // and 'delete' actions
19  'actions'=&gt;array('admin','delete'),
20  'users'=&gt;array('admin'),
21  ),
22  array('deny', // deny all users
23  'users'=&gt;array('*'),
24  ),
25  );
26  }
27 
28 </code></div>
29 
30 <p>With your new filter you will have new rule sets that you can use by themeselves or combined togheter with both the new and old rules, without having to learn
31 a new data structure for writing them.</p>
32 <p>Now we'll be analyzing every single new type of rule that you can apply and will show you an example for each one of them.</p>
33 <p>Keep in mind that all of the results achieved by this new kind of rules could have been obtained using the good old expression rule.<br/>
34 The rules we provide however make it easier for you to write them, and furthermore will be much more easy to read.</p>

36 <h3>Ajax!</h3>
37 <p>With this rule you can force the user to load a specific action with or without ajax.</p>
38 
39 <div class="code"><code>
40  <b>// allow all users to perform 'index' and 'view' actions
41  // just when loading the page via ajax</b>
42  array('allow',
43  'actions'=&gt;array('index','view'),
44  'users'=&gt;array('*'),
45  <span class="highlight">'ajax'=&gt;true,</span>
46  ),
47 
48 </code></div>
49 <p>If you want to force the user to load a page without ajax you'll just have to assign to it a <b>boolean false</b></p>
50 <h3>Groups, of course!</h3>
51 <p>This rule allow you to choose which groups of users have access to a specific action.<br/>
52 This rule behaves just like the well known users rule:</p>
53 
54 <div class="code"><code>
55  <b>// allow all users who belong to the admin or core group
56  // or to the group with id 5 to perform 'update' and 'admin' actions</b>
57  array('allow',
58  'actions'=&gt;array('update','admin'),
59  <span class="highlight">'groups'=&gt;array('admin', 'core', 5),</span>
60  ),
61 
62 </code></div>
63 <p>You can also use the <b>*</b> to indicate that you grant access to every group.<br/>
64 If this rule is setted guests won't have of course access to the page.</p>

66 <h3>Levels</h3>
67 <p>Each group has a level assigned, and so each user inherits the level of the group he belongs to.<br/>
68 You will be able to decide the level of each group, but will talk more about it in the <b>Root Tools section of the documentation.</b></p>
69 
70 <p>This is one of the most complex rules you can apply. You can tell the system that just users with a level higher then 4 can
71 execute a specific action, or you can decide that users with a level equal to 5 or higher then 50 can view the page, or
72 if you really want to you can decide that just those who match all the above defined rules can access that content on your
73 application.<br/>
74 Let's see some examples to make it more clear:</p>
75 
76 <div class="code"><code>
77  <b>// allow all users with a level lower then 10, or equal to 20
78  // or higher or equal to 35 to perform an update action</b>
79  array('allow',
80  'actions'=&gt;array('update'),
81  <span class="highlight">'level'=&gt;array('&lt;10', '20' ,'&gt;=35'),</span>
82  ),
83 
84 </code></div>
85 <div class="code"><code>
86  <b>// allow all users with a level lower then 40 AND
87  // higher then 35 to perform an update action</b>
88  array('allow',
89  'actions'=&gt;array('update'),
90  <span class="highlight">'level'=&gt;array('&lt;40', '&gt;35', 'strict' =&gt; true),</span>
91  ),
92 
93 </code></div>
94 <p>I think both example are self explenatory. You can use any comparison operator, but remember to <b>never use =</b> because it will always return true.</p>
95 <p>If you are using this rule guests won't have access to the page, because they don't have a level.</p>

97 <h3>P-what? PBAC! Permission Based Access Control.</h3>
98 <p>userGroups lets you manage groups and users, but also their permission.<br/>
99 For each controller you will be able to assign to every user and/or group reading, writing, and admin permissions.<br/>
100 A user authomatically inherits his group permissions, but if you want to you can furthermore customize adding special rules to him.<br/>
101 We'll talk about assigning permissions in the <b>Root Tools</b> chapter of the documentation, so if you want more infos about it take a look overthere, but
102 right now what we already told you is enough to understand how to use this kind of rule.</p>
103 <p>With PBAC you'll be able to grant access to a page just to users with writing permissions on that controller or admin permission on another.</p>
104 <p>Let's make it simpler. You have in your applications two controllers, one that is managing companies, the other one employees.<br/>
105 Just users with admin permission on companies can delete them, so you write this rule:</p>
106 
107 <div class="code"><code>
108  <b>// allow users with admin permissions to delete companies</b>
109  array('allow',
110  'actions'=&gt;array('delete'),
111  <span class="highlight">'pbac'=&gt;array('admin'),</span>
112  ),
113 
114 </code></div>
115 
116 <p>Easy, wasn't it? In the employee controller you want to grand access to the update action both to users with admin rights on the employee controller and
117 to users with admin rights on the companies controller</p>
118 
119 <div class="code"><code>
120  <b>// allow users with admin permissions on this controller or
121  // the companies controller to access the update action.</b>
122  array('allow',
123  'actions'=&gt;array('update'),
124  <span class="highlight">'pbac'=&gt;array('admin', 'companies.admin'),</span>
125  ),
126 
127 </code></div>
128 
129 <p>In our application we have a installed a module that manage employees certifications named <b>certEmployee</b>.<br/>
130 We want those users with admin permissions on that module user controller to have access to our admin
131 action on the employee controller, but we don't want them to access that page using ajax.<br/>
132 So we write the following rule inside our employee controller:</p>
133 
134 <div class="code"><code>
135  <b>// allow users with admin permissions on the user controller that
136  // belongs to the certEmployee module to perform update actions, but
137  // just if they access that page without using ajax</b>
138  array('allow',
139  'actions'=&gt;array('update'),
140  <span class="highlight">'pbac'=&gt;array('certEmployee.user.admin'),</span>
141  <span class="highlight">'ajax'=&gt;false,</span>
142  ),
143 
144 </code></div>
145 
146 <p>As you can see PBAC uses a dot notation. If you don't use any dot PBAC will check the user permissions on that controller, if you use one dot, the
147 word before it will be the controller name, if you use two dots the first word will be the module name.<br/>
148 Controllers in the basic applications are considered inside a module named Basic. So if you want to check the user admin permission on the site
149 controller that comes by default with a fresh new Yii application you'll have to write "<b>Basic.site.admin</b>"</p>
150 <p>Remember that PBAC is <b>case sensitive</b>. If you have doubts about how to write the name of your controller check its web path, print the
151 value of <b>Yii::app()-&gt;getController()-&gt;id</b>, or look at it's name on the <b>Root Tools</b> page.</p>
152 
153 <p>If you use PBAC no guest user will have access to that page, because of course guest don't have any permission.</p>

176 <h3>And what abou PBAC? Can I check permissions?</h3>
177 <p>Of course you can, but checking permissions could have been a too cumbersome task to perform, so we provided you a special method:</p>
178 <p><b>Yii::app()-&gt;user-&gt;pbac();</b></p>
179 <p>the user::pbac() method can accept up to 3 arguments, and just the first one is required.</p>
180 <dl>
181 <dt>1st argument: permission (mixed)</dt>
182  <dd>
183  This is the permission you need to check. You can send both a string or an array. You'll be able to use the same dot notation that
184  you used in your rules.<br/>
185  If you just need to check one permission you can just send a string, but if you need to check more then one
186  you'll need to send an array, where each value is a permission to check (just like in the rule).<br/>
187  Remember that when this method is used if you don't specifically tell the controller or module name the method will use those of the
188  page that the user is browsing.
189  </dd>
190 <dt>2nd argument: configuration (string)</dt>
191  <dd>
192  userGroups has several configurations, if you want to check at the same time both a configuration value and a permission to decide
193  whenever a user can view a portion of your page you can input the configuration name inside this argument.
194  </dd>
195 <dt>3rd argument: operator ('AND' or 'OR')</dt>
196  <dd>
197  Sometimes you may want to grant access to a link if the user has a specific permission OR if that configuration is active.<br/>
198  By default the user::pbac() method assumes that you are using an AND operator, so if you need to you can pass the string OR to this
199  argument to change the comparison.
200  </dd>
201 </dl>

203 <h3>There's a new User in Town!</h3>
204 <p>You well know that in Yii you can target every user using a <b>*</b>, guests using a <b>?</b> and registered users using a <b>@</b>.</p>
205 <p>We added a new user tipology with this module: <b>#</b><br/>
206 <b>#</b> identifies users in recovery mode. Users are in recovery mode when they are activating their accounts or resetting their password.
207 Users in recovery mode can access just pages where guests can go and those that are specifically aimed to them.<br/>
208 No other user, not even guest can access pages reserved for the recovery mode users.</p>
209 
210 <h3>Who's your Daddy? ROOT!</h3>
211 <p>When you installed this module you created on the fly a new user, the one you are now logged in as. That user is known as Root.<br/>
212 Root cannot be banned, his permissions cannot be changed, no one but himself can change his profile, and has automatically access granted to
213 every single page in the application, but those reserved to recovery mode users.</p>
214 <p>You'll never have to specifically tell any rule to grant access to ROOT, because he will always match any condition you write.</p>
