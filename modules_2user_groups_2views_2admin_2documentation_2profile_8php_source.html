6 <h3>First Step: create the database table and model</h3>
7 <p>Create a database table just like you where always used to do.<br/>
8 Just remember to insert a column named <b>ug_id</b>, make it a bigint with length of 20 and unique.</p>
9 
10 <p>Right after that create your model, you can do it manually or using gii if you prefer.</p>

12 <h3>Adding the profileViews method to the class</h3>
13 <p>To work properly userGroups needs that you add a new method to your model: <b>profileViews</b></p>
14 
15 <div class="code"><code>
20  public function profileViews()
21  {
22  return array(
23  UserGroupsUser::VIEW =&gt; 'index',
24  UserGroupsUser::EDIT =&gt; 'update',
25  UserGroupsUser::REGISTRATION =&gt; 'registration',
26  );
27  }
28 
29 </code></div>

31 <p>profileViews will be called by the userGroups module every single time it will try to render a profile view.<br/>
32 Let's say that a user goes to his page profile.<br/>
33 userGroups will look in the list of models that extend the user profile, looking for those that are returning a value for the
34 <b>userGroupsUser::VIEW</b> key.<br/>
35 If the model returns a value for that key, userGroups will load the view named after that value.</p>
36 
37 <p>To let userGroups find your view file you'll have to put it at this path:<br/>
38 <b>protected/views/TABLENAME/</b><br/>
39 
40 So if you created a new model for the table <i>user_hobbies</i>, userGroups will look for the view file inside<br/>
41 <b>protected/views/user_hobbies/</b></p>
42 
43 <h3>Creating UserGroupsUser::VIEW views</h3>
44 <p>View pages are loaded when someone loads a profile.<br/>
45 In your view file related to UserGroupsUser::VIEW you will have a <b>$model</b> variable that will contain your Profile Extension model.</p>
46

48 <h3>Creating UserGroupsUser::UPDATE views</h3>
49 <p>In the update views you'll still have your profile extension model inside the <b>$model</b> variable.<br/>
50 You will also have a variable named <b>$user_id</b> containing the user ID of the profile that is being edited.</p>
51 <p>In your update view you can use ajax validation and we encourage you to do so, but you have to pay attention to the submit button.<br/>
52 Here you can see an example of submit button:</p>
53 
54 <div class="code"><code>
55  echo CHtml::ajaxSubmitButton(
56  Yii::t('userGroupsModule.general','Update External Profile'), // first line
57  Yii::app()-&gt;baseUrl . '/userGroups/user/update/id/'.$user_id, // second line
58  array('update' =&gt; '#userGroups-container'), // third line
59  array('id' =&gt; 'submit-profile-'.$model-&gt;id.rand()) // fourth line
60  );
61 
62 </code></div>
63 
64 <p>In the first line we are using Yii::t() method to define the name of the submit button. You don't have to do so if you don't
65 care about localization, so you can use a string here instead.<br/>
66 In the second line we are setting the path of the form. Make sure you right it down exactly as it is, unless you want to use
67 your own controller to deal with the saving action. In that case point the form to the correct path.<br/>
68 In the third line we set the javascript action to perform right after the form execution. Copy it exactly as it is.<br/>
69 In the fourth line we set a random id name for the button. That's an important thing to do if you don't want to have some
70 ajax overloading.</p>

72 <h3>Creating UserGroupsUser::REGISTRATION views</h3>
73 <p>In the registration view files you'll just have to insert your form inputs and nothing else.<br/>
74 Your inputs will be part of the registration form and userGroups will take care of validation and storing the data on the database.<br/>
75 In this views you'll have your Profile Extension model inside the <b>$model</b> variable.</p>
76 
77 <h3>So i can use normal model validation and whatsoever?</h3>
78 <p>Of course you can, and you can use whatever other model features you are used to, like afterFind, beforeSave, and so on.<br/>
79 Right now userGroups supports profile extension just in profile view and editing actions, not on registration or recovery.<br/>
80 Anyway we encourage you to use rules based on scenarios, to avoid future problems.<br/>
81 The scenario used on the UserGroupsUser::UPDATE views is: <b>updateProfile</b>.<br/>
82 The scenario used on the UserGroupsUser::REGISTRATION view is <b>registration</b>.</p>

85 <h3>So, after I set all this stuff what do i have to do to make userGroups aware of my Profile Extension?</h3>
86 <p>That, as usual, is pretty easy: open your application config file and add the profile paramter to the userGroups module.<br/>
87 This parameter must contain an array where every value is the name of the model class that you are going to include.</p>
88

90  'userGroups'=&gt;array(
91  'accessCode'=&gt;'your access code, by now useless',
92  <span class="highlight">'profile' =&gt; array('UserHobbies'),</span>
93  )
94 
95 </code></div>

97 <h3>Is Profile Extensions using the relations system of CActiveRecord?</h3>
98 <p>Of course it is, it's forging HAS_ONE relations on the fly, so if you happen to use the UserGroupsUser model you can find all the data of
99 your Profile Extensions, just using this syntax:<br/>
100 <b>$model-&gt;relMODEL_CLASS_NAME</b><br/>
101 So, in the UserHobbies example we used until now this would be the right syntax:
102 <b>$model-&gt;relUserHobbies</b>
